오늘의 핵심 질문

스마트컨트랙트 보안은 어려운 해킹 기술부터 외우는 과목이 아니다. 먼저 함수 권한, 상태 변경 순서, 외부 호출 위치를 보는 습관이 필요하다.

1. 이 함수는 누구에게 열려 있는가?

`public` 함수는 모두가 호출할 수 있다. 테스트용으로 열어 둔 초기화 함수가 배포 후에는 지갑 탈취 경로가 될 수 있다.

2. 장부를 먼저 고쳤는가?

ETH를 먼저 보내고 잔액을 나중에 줄이면, 외부 컨트랙트가 같은 출금 함수를 다시 부를 수 있다.

3. delegatecall은 누구의 storage를 바꾸는가?

다른 컨트랙트의 코드를 빌려오더라도 저장소는 호출자 쪽 storage가 바뀐다. Parity 1차에서는 열린 초기화/권한 경로가 이 구조와 결합해 Wallet owner 상태를 바꿀 수 있었다.

한 줄 요약: 핵심을 단순화하면 The DAO는 "외부 호출 전에 상태를 확정하지 않은 문제"이고, Parity Wallet은 "초기화 함수와 라이브러리 권한 관리를 잘못한 문제"이다.
1

먼저 알아야 할 기본 개념

시뮬레이션을 이해하려면 상태값, 외부 호출, 초기화 함수, delegatecall의 의미를 먼저 잡아야 한다.

상태값은 컨트랙트가 기억하는 장부다

`balances[msg.sender]`는 지금 함수를 부른 사람의 잔액이고, `owner`는 관리자 주소다. 스마트컨트랙트 보안에서는 이 장부가 언제 바뀌는지가 중요하다.

mapping(address => uint256) public balances;
address public owner;
bool public initialized;

외부 호출은 제어권을 잠깐 넘기는 일이다

ETH를 보내는 코드는 단순 송금처럼 보이지만, 상대방이 컨트랙트라면 그 컨트랙트의 `receive()` 함수가 실행될 수 있다.

msg.sender.call{value: amount}("");

초기화 함수는 배포 후 constructor처럼 쓰인다

프록시나 라이브러리 구조에서는 `initWallet()` 같은 함수를 따로 둔다. 반드시 한 번만 실행되어야 하고, 아무나 호출할 수 없어야 한다.

delegatecall은 코드만 빌려 쓰는 호출이다

라이브러리 코드가 실행되지만 storage는 지갑 컨트랙트 쪽이 바뀐다. 열린 initializer나 권한 함수가 있으면 Wallet의 owner 상태가 바뀔 수 있다.

사건 1: The DAO 해킹, 2016

The DAO는 이더리움 위의 탈중앙화 투자 조직이었다. 실제 사건은 splitDAO 재귀 호출을 통해 대량의 ETH가 분리된 child DAO로 이동한 사례였고, 이후 이더리움 하드포크 논쟁으로 이어졌다. 아래 ATM 모델은 그 재진입 원리를 단순화한 도식이다.

날짜 정리: 2016년 6월 The DAO 공격 및 취약점 공지, 2016년 7월 하드포크 논의로 이어짐 사건이다.

위험한 순서

function withdraw(uint256 amount) public {
  require(balances[msg.sender] >= amount);
  msg.sender.call{value: amount}("");
  balances[msg.sender] -= amount;
}

핵심을 단순화하면, 외부로 제어권을 넘긴 뒤 장부를 나중에 고치는 순서가 문제다. 외부 호출 사이에 공격자가 다시 들어올 수 있다.

안전한 순서

function withdraw(uint256 amount) public {
  require(balances[msg.sender] >= amount);
  balances[msg.sender] -= amount;
  (bool ok, ) = msg.sender.call{value: amount}("");
  require(ok);
}

조건 확인, 상태 변경, 외부 호출 순서다. 이것이 Checks-Effects-Interactions 패턴이며, 실제 구현에서는 call 결과 확인과 ReentrancyGuard 또는 pull-payment 패턴도 함께 검토한다.

사건 2: Parity Wallet 1차 해킹

Parity 멀티시그 지갑은 라이브러리 컨트랙트에 기능을 위임했다. 문제는 초기화 함수가 보호되지 않아 공격자가 자신을 owner로 등록할 수 있었다는 점이다.

날짜 정리: 2017년 7월 1차 사건은 initWallet 미보호로 owner가 탈취된 사건이다.

초기화 함수 미보호

`initWallet()`이 배포 후에도 다시 호출 가능하고 initialized guard나 접근제어가 없으면 공격자가 owner 목록을 바꿀 수 있다. 실제 Parity 사고는 단일 owner 변수만이 아니라 multi-sig owner 배열과 threshold 계열 권한 상태가 바뀐 사례다.

delegatecall 구조

라이브러리 코드는 지갑 storage context에서 실행된다. 따라서 라이브러리의 열린 initializer나 권한 함수가 지갑 권한 실수로 이어질 수 있다.

결과

공격자가 owner가 된 뒤 멀티시그 지갑의 자금을 전송할 수 있었다.

시뮬레이션 C: delegatecall storage 지도

delegatecall은 "라이브러리 코드 실행"이지만 storage는 "지갑 storage"를 사용한다. 아래 그림에서 owner 슬롯이 어디서 바뀌는지 확인한다.

코드는 Library, 장부는 Wallet

버튼을 누르면 실행 화살표가 움직이고 바뀌는 storage 슬롯이 강조된다.

대기
사용 방법실행 전 canvas에서 Library와 Wallet을 구분한 뒤 delegatecall 실행을 누르고, 실행 후 owner 슬롯 위치를 다시 본다.
관찰 포인트코드는 Library에서 선택되지만 storage 쓰기는 Wallet slot에 발생한다. 로그의 "Library storage가 아님" 문장을 확인한다.
왜 필요한가Parity 1차의 핵심은 열린 initializer/권한 경로가 delegatecall을 통해 Wallet 상태를 바꿀 수 있었다는 점이다.
실행 코드Library code is selected by delegatecall.
저장공간Wallet storage is still the write target.
위험 원인Wallet이 Library의 열린 initializer나 권한 함수를 delegatecall로 실행하면, Library 코드가 Wallet owner 상태를 바꿀 수 있다.
방어 원리initializer를 배포 과정에서 한 번만 실행하고, 재초기화와 직접 호출 경로를 막는다. storage layout 검사는 일반 delegatecall 위험 검토에 포함한다.

시뮬레이션 D: Parity 2차 자금 동결

여러 지갑이 하나의 라이브러리에 의존하면, 라이브러리 하나가 사라졌을 때 여러 지갑이 동시에 기능을 잃는다.

날짜 정리: 2017년 11월 2차 사건은 공유 라이브러리 코드가 삭제되어 여러 지갑의 자금이 동결된 사건이다.
현재 EVM과의 차이: 이 시뮬레이션은 2017년 당시 Parity 사고의 SELFDESTRUCT 동작을 설명한다. post-Cancun EVM에서는 EIP-6780에 따라 같은 트랜잭션에서 생성된 컨트랙트 예외를 제외하면 SELFDESTRUCT가 기존 컨트랙트의 코드, storage, 계정을 삭제하지 않는다. 단, 계정 잔액을 beneficiary로 전송하는 동작은 남는다. 여기서는 역사적 사건의 원인 이해에 초점을 둔다.

공유 라이브러리 selfdestruct 효과

라이브러리 초기화 → owner 획득 → kill 실행 → 지갑 기능 마비 순서를 재연한다.

관찰 포인트ETH balance는 남아 있고, library code path와 withdraw execution만 끊긴다. 돈이 사라진 사건이 아니라 실행 경로가 막힌 사건이다.

용어 정리와 참고자료

Reentrancy

함수가 끝나기 전에 외부 컨트랙트가 같은 함수를 다시 호출하는 공격 방식.

Initializer

배포 후 owner나 기본 설정을 넣는 함수. public/external일 수 있지만 반드시 한 번만 실행되고 재초기화가 막혀야 한다.

delegatecall

다른 컨트랙트의 코드를 빌려 실행하지만 storage는 호출자 쪽을 쓰는 저수준 호출.

교육용 단순화: 이 자료는 사건의 핵심 보안 원리를 설명하기 위한 단순화 모델이며, 실제 감사·투자·법률 조언이나 최신 메인넷 동작 보증이 아니다. 실무 적용 전 최신 Solidity, EVM, 라이브러리 문서와 전문 감사를 확인해야 한다.
